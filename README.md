# vLLM Diagnose Prediction

A beginner-friendly, stable repository for performing diagnosis (ICD) prediction tasks on the MIMIC dataset using modern Large Language Models (LLMs) via `vLLM`. 

This repository is designed as a foundational template for master's students to build upon. It provides:
- A scalable architecture deploying the LLM on a Kubernetes cluster.
- Efficient data loading for Parquet-based datasets (like MIMIC).
- A stable, async inference pipeline using the OpenAI API spec.
- Guided decoding for multi-label ICD classification using Pydantic JSON schemas to guarantee parseable outputs.
- Evaluation metrics (Precision, Recall, F1).

---

## 🚀 Quick Start Guide for Students

This guide will walk you through building your client, configuring your environment, and deploying the job on the Kubernetes cluster.

### 1. Structure Overview
- `configs/config.yaml`: Your central control panel. Change model endpoints, dataset paths, limits, and generation variables here.
- `data/mimic/`: Place your clinical MIMIC dataset files here. Note: this folder is ignored by git to protect sensitive patient data.
- `scripts/`: Utility scripts to build Docker containers and generate Kubernetes YAMLs.
- `src/`: The core python code for loading data, prompting the LLM, parsing responses, and scoring accuracy.
- `k8s/`: Kubernetes manifests to deploy your containers (autogenerated using scripts).
- `Dockerfile`: Instructions to package your code so it can run isolated on the cluster.

### 2. Local Setup & Data Placement

Before deploying, set up your python environment and pull in the MIMIC dataset.

```bash
# Create a virtual environment
python -m venv venv
source venv/bin/activate

# Install dependencies
pip install -r requirements.txt
```

**MIMIC Dataset:** Ensure you copy your specific MIMIC patient file (e.g., `patients_1411.pq`) into the `data/mimic/` folder and update your config so `configs/config.yaml` points to it properly:
`patients_file: "data/mimic/mimic_dataset.pq"`

### 3. Containerizing Your Code

To run your inference on the Kubernetes cluster, you need to package it into a Docker image. Previously, this required manual docker commands, but now you can just use the provided build script!

1. Update your image tag in `configs/config.yaml` under `k8s: client: image`. Make sure it points to your accessible registry.
2. Run the automated build script:
   ```bash
   python src/build_docker.py --config configs/config.yaml
   ```
   *This single command builds your container using the `Dockerfile` and pushes it to the registry. It requires you to be logged in to Docker locally (`docker login`).*

### 4. Deploying to Kubernetes

With your code packaged, you deploy it using the files in the `k8s/` directory.

#### Generating Kubernetes Manifests
Instead of editing yaml files by hand, you configure everything in `configs/config.yaml` and generate the manifest files automatically!

1. Update `configs/config.yaml` with your image name, GPU requirements, model size, etc. under the `k8s` and `model` sections.
2. Run the deployment generator:
   ```bash
   python scripts/deploy.py --config configs/config.yaml
   ```
3. This creates or updates `k8s/vllm_server.yaml` and `k8s/inference_job.yaml` securely configured for your run.

#### Starting the vLLM Server (If not already running)
To host the LLM weights and serve requests, you need a server alive.

1. Verify the Huggingface token secret exists on your cluster.
2. Deploy the server:
   ```bash
   kubectl apply -f k8s/vllm_server.yaml
   ```
3. Wait for it to become ready:
   ```bash
   kubectl get pods -l app=vllm-server -w
   ```
   *(Note: Models are huge; it may take ~10 minutes to download weights if not cached).*

#### Running Your Inference Job

The pipeline itself is run as a Kubernetes "Job".

1. Ensure your cluster has a PersistentVolumeClaim (PVC) matching `claimName: data-volume` where the MIMIC data lives.
2. Deploy the job:
   ```bash
   kubectl apply -f k8s/inference_job.yaml
   ```

### 5. Monitoring & Results

Watch your job run its course:
```bash
# See the job spin up
kubectl get jobs

# Watch the logs of the inference script
# Replace 'diagnose-prediction-client-xxxxx' with your actual pod name attached to the job
kubectl logs -f pod/diagnose-prediction-client-xxxxx
```

Once completed, the script saves your processed predictions (`*_predictions.csv`) in the directory specified by your configured data files. You can retrieve these using `kubectl cp` or by mapping a shared storage volume.

### Tips for Extensions

- **Updating the Schema:** Want the model to extract Symptoms before Diagnoses? Edit the `ICDPrediction` class in `src/prompter.py`. `vLLM` will automatically enforce your new JSON structure!
- **Few-Shot Prompting:** Edit `src/prompter.py` to pass example mappings from clinical-note to ICDs inside the prompt template.
